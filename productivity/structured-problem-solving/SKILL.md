# Structured Problem Solving

## Overview
This umbrella skill consolidates generic planning/problem-solving helpers that were previously split into separate micro-skills.

## Core workflow
1. Decide whether the task needs exploration, decomposition, execution tracking, or persistent visibility.
2. Start with the lightest structure that fits the task.
3. Keep one active next step at a time even when the plan is broad.
4. Re-plan when new information invalidates assumptions.

## Consolidated subsections
### Exploratory questioning
Use a questioning-first mode when the goal is to understand the real problem before proposing answers.

### Reduce-rank / root-drill mode
When the user asks for the essence of a field, phenomenon, decision, or recurring blocker, use a two-axis lens inspired by ljg-skills instead of making a broad list:

- **Reduce rank**: find the few independent generators that can reproduce the visible symptoms. Ask: what forces, constraints, incentives, or feedback loops generate most of this domain? Stop when removing one generator makes the explanation unable to rebuild the observed cases.
- **Root drill**: for one claim or pain point, keep moving downward from surface event → repeated pattern → hidden assumption → irreducible tension → actionable sentence.

Use reduce-rank for broad domains (`考研复习系统的秩是什么`, `AI Agent 成本控制靠哪几根力撑着`); use root-drill for one sharp question (`为什么我总卡在开始`, `这个主页为什么不能写满`). Output should end with one compact judgment and one next action, not just a taxonomy.

### Decomposition and action planning
Break a large problem into concrete steps, dependencies, and priorities.

### Local to-do tracking
Use a lightweight checklist or session plan when persistent external tracking is unnecessary.

### Persistent task visibility
Use an external task/memory system when the work spans time, collaborators, or remote monitoring.

## Common pitfalls
- Solving before clarifying.
- Creating elaborate plans for trivially small tasks.
- Tracking tasks in two systems without a source of truth.
- Leaving broad plans without a clearly active next step.

## Verification checklist
- [ ] Exploration vs execution need identified.
- [ ] Task broken into actionable next steps when needed.
- [ ] One tracking system chosen as source of truth.
- [ ] Plan updated after new information or blockers.
